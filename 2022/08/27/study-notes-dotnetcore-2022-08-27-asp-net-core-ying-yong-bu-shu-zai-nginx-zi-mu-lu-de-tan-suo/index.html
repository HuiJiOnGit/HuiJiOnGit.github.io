asp.net core 应用部署在Nginx 子目录的探索

aspnetcore nginx
aspnetcore
发布日期:   2022-08-27

起因

由于没有测试用的域名, 一个项目部署到了已有域名的子目录下, 用的Nginx反向代理, 访问链接是这样

http://test.com/app

然后导致的问题就是页面能访问到了,但是像 js 和 css 的引入全部都 404 了,因为这些文件都是这样引入的

<link rel="stylesheet" href="~/lib/bootstrap/dist/css/bootstrap.min.css" />

假设现在的域名是test.com,那么现在文件的请求都是

http://test.com/lib/bootstrap/dist/css/bootstrap.min.css

直接全部 404 了,领导解决这个问题的方式很麻烦,他将所有的这些引用全部修改为了

<link rel="stylesheet" href="@Model.Host/lib/bootstrap/dist/css/bootstrap.min.css" />

这个Host就是保存在配置的要使用的路径http://test.com/lib

高情商: 这是一个很直接了当的修改方案,大幅提高工作量

低情商: Rubbish

解决方案

先展示下最开始的nginx的配置

server&#123;
  listen 53346;
  server_name localhost;
   location /app/&#123;
    proxy_pass  http://localhost:5243/;
    proxy_http_version 1.1;
    proxy_set_header Upgrade $http_upgrade;
    proxy_set_header Connection keep-alive;
    proxy_set_header Host $host:$server_port;
    proxy_cache_bypass $http_upgrade;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
            index  index.html index.htm;
        &#125;
        location = /50x.html &#123;
            root   html;
        &#125;
 &#125;

第一个问题

加上子目录之后的路径访问会 404, 为什么呢? 很简单,因为加上子目录之后, 路由对不上呢, 比如你要访问 /Home/Index,但是实际访问却是/app/Home/Index,这样改是找不到路由的

搜索一下之后发现, asp.net core 中的PathBase就是设计出来应对这种情况的,就是告诉应用, 我这个请求前面是有一段不算在路由里面的, 你找路由要先排除这个.在中间件开始的地方加上

app.UsePathBase(new PathString("/app"));

第二个问题

但是访问之后发现, 路由时找到了,但是 js 和 css 都是 404,看控制台发现,这里的请求并没有带上/app,而是直接访问了域名,所以还要调整下,在nginx那里要把/app路径传入进去


# proxy_pass  http://localhost:5243/;

proxy_pass  http://localhost:5243/app/;

重启nginx之后,访问果然正确显示引用的 js 和 css 了,路径已经加上/app了

最后贴下完整的配置

nginx 配置

  server_name localhost;

要注意的点

  • 链接

如果你使用了 asp-action asp-router asp-controller 等TagHelper去生成链接的话, 那就会带上子目录的路径的, 如果你直接写一个 a 标签,那这个在代理之后生成的是不会有子目录,所以涉及这个最好使用tagheler来生成.

<a asp-page="/Privacy">使用TagHelper生成的</a>
<!--生成是这样的<a href="/app/Privacy">使用TagHelper生成的</a>-->
<a href="/Private">手动写链接的</a>
<!--生成如下<a href="/Private">手动写链接的</a>-->
<a href="~/Private">手动写链接的,但是使用了~</a>
<!--<a href="/app/Private">手动写链接的,但是使用了~</a>-->

建议使用 1,3,不要用 2

  • 请求

那 js 请求的话,也是同样道理的,建议使用axios来发送请求,可以轻松的设置baseURL来配置子目录,

说到这个,不得不提一嘴同事写的解决子目录ajax请求的方法,直接替换了jQuery的发送请求方法, 保存原本的方法在临时变量中, 然后修改请求的 url, 再用保存在临时变量中的方法去发送真正的请求, 不是说不行, 而是jQuery没有这种对应基础配置,只能用这种迂回的方式, 不太好用, 现在jQuery是真的落后了

  • 其他资源(音频视频)

这里有两种引入方式,会生成不一样的结果

<img src="/img/001.jpg" asp-append-version />
<!--<img src="/img/001.jpg?v=LWv2Dubq1qrHEyLduWAtYgtEtCVlddyQv-omxaAMgiI">-->
<img src="~/img/001.jpg" asp-append-version />
<!--<img src="/app/img/001.jpg?v=LWv2Dubq1qrHEyLduWAtYgtEtCVlddyQv-omxaAMgiI">-->

使用后者可以自动添加子目录,推荐使用后者

另外探究下源码这里是怎么处理PathBase的,首先看看taghelper是怎么处理的,一路顺着源码走下去发现以下调用

AnchorTagHelper
这里无论使用使用路由,action,还是page都是使用了UrlHelperFactory类型的方法,接下看看这个

public override void Process(TagHelperContext context, TagHelperOutput output)
{
    // 省略了一部分验证
    var routeLink = Route != null;
    var actionLink = Controller != null || Action != null;
    var pageLink = Page != null || PageHandler != null;

    if ((routeLink && actionLink) || (routeLink && pageLink) || (actionLink && pageLink))
    {
        var message = string.Join(
            Environment.NewLine,
            Resources.FormatCannotDetermineAttributeFor(Href, ""),
            RouteAttributeName,
            ControllerAttributeName + ", " + ActionAttributeName,
            PageAttributeName + ", " + PageHandlerAttributeName);

        throw new InvalidOperationException(message);
    }

    RouteValueDictionary routeValues = null;
    if (_routeValues != null && _routeValues.Count > 0)
    {
        routeValues = new RouteValueDictionary(_routeValues);
    }

    if (Area != null)
    {
        // Unconditionally replace any value from asp-route-area.
        if (routeValues == null)
        {
            routeValues = new RouteValueDictionary();
        }
        routeValues["area"] = Area;
    }

    TagBuilder tagBuilder;
    if (pageLink)
    {
        tagBuilder = Generator.GeneratePageLink(
            ViewContext,
            linkText: string.Empty,
            pageName: Page,
            pageHandler: PageHandler,
            protocol: Protocol,
            hostname: Host,
            fragment: Fragment,
            routeValues: routeValues,
            htmlAttributes: null);
    }
    else if (routeLink)
    {
        tagBuilder = Generator.GenerateRouteLink(
            ViewContext,
            linkText: string.Empty,
            routeName: Route,
            protocol: Protocol,
            hostName: Host,
            fragment: Fragment,
            routeValues: routeValues,
            htmlAttributes: null);
    }
    else
    {
        tagBuilder = Generator.GenerateActionLink(
            ViewContext,
            linkText: string.Empty,
            actionName: Action,
            controllerName: Controller,
            protocol: Protocol,
            hostname: Host,
            fragment: Fragment,
            routeValues: routeValues,
            htmlAttributes: null);
    }

    output.MergeAttributes(tagBuilder);
}

UrlHelperFactory
然后这里有个分支,有两个实现了IUrlHelper的类型,一个是EndpointRoutingUrlHelper,另外一个是UrlHelper,由于没有调试,只能两个都看看,

public IUrlHelper GetUrlHelper(ActionContext context)
{
    // 省略了一部分验证

    var endpoint = httpContext.GetEndpoint();
    if (endpoint != null)
    {
        var services = httpContext.RequestServices;
        var linkGenerator = services.GetRequiredService();
        var logger = services.GetRequiredService>();

        urlHelper = new EndpointRoutingUrlHelper(
            context,
            linkGenerator,
            logger);
    }
    else
    {
        urlHelper = new UrlHelper(context);
    }

    httpContext.Items[typeof(IUrlHelper)] = urlHelper;

    return urlHelper;
}

最后看到其实两个IUrlHelper都是使用了UrlHelperBase父类中的GenerateUrl方法,然后这两个方法都会调用到AppendPathAndFragment这个方法,

internal static void AppendPathAndFragment(StringBuilder builder, PathString pathBase, string virtualPath, string? fragment)
{
    if (!pathBase.HasValue)
    {
        if (virtualPath.Length == 0)
        {
            builder.Append('/');
        }
        else
        {
            if (!virtualPath.StartsWith('/'))
            {
                builder.Append('/');
            }

            builder.Append(virtualPath);
        }
    }
    else
    {
        if (virtualPath.Length == 0)
        {
            builder.Append(pathBase.Value);
        }
        else
        {
            builder.Append(pathBase.Value);

            if (pathBase.Value.EndsWith("/", StringComparison.Ordinal))
            {
                builder.Length--;
            }

            if (!virtualPath.StartsWith("/", StringComparison.Ordinal))
            {
                builder.Append('/');
            }

            builder.Append(virtualPath);
        }
    }

    if (!string.IsNullOrEmpty(fragment))
    {
        builder.Append('#').Append(fragment);
    }
}

至此,TagHelper是怎么处理PathBase已完全解开秘密.

还有个疑问,为什么用~这个符号开始的链接也能正确显示?

大概猜测是在UrlHelperBase的TryFastGenerateUrl中,还不是很确定

总结

通过nginx的配置就完全可以解决部署在子目录中的问题, 部署的问题就应该交给反向代理方处理而不是硬改代码,代码应尽量少改才对.

还是就是我上面是测试用,所以子目录在代码中是写死, 实际应用中应该写到配置中可以灵活应用

另外就是要注意 nginx 中的代理和转发最后都要有/,不然点击链接会出现跳转 404
